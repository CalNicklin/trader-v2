# Debate: Where Should Population Recovery Logic Live?

## Context

The trading system's `gap_fade_v1` strategy was killed by the daily tournament (15.92% max drawdown). Population dropped from 3 to 2 strategies. Neither remaining strategy has 30+ trades, so the weekly evolution cycle will skip — the system is stuck.

**Three proposed approaches:**

- **A) VPS-side evolution** — Add a population floor to `src/evolution/index.ts` that bypasses the 30-trade requirement when population is critically low, letting Claude propose structural mutations without trade data
- **B) GitHub Actions self-improvement agent** — Expand the `claude.yml` prompt so the Opus agent recognises low population as a priority and creates PRs adding new seed strategies or adjusting thresholds
- **C) Both** — VPS evolution gets the population floor for immediate response; self-improvement agent gets population awareness for longer-term fixes

**Key constraints:**
- Evolution runs Sunday 18:00 UK (VPS, Claude API call)
- Self-improvement runs Sunday 19:00 UK (GitHub Actions, Opus agent with full code access)
- Evolution needs 30+ trades to propose mutations (currently blocks on low-trade strategies)
- Self-improvement can modify any code, create PRs, but changes deploy via CI (not instant)
- Budget: Haiku cheap, Sonnet standard, Opus expensive
- Current population: 2/8 strategies, both with <2 trades

## Rules
- Each advocate argues for their approach across 5 rounds
- Advocates should engage with and respond to the others' arguments
- Final round: each advocate must state whether they've changed their position

## Round 1: Opening Arguments

### Advocate A (VPS-side evolution)

Population recovery is fundamentally a runtime self-healing concern, not a code-change concern. The system already has all the machinery it needs: the evolution cycle in `src/evolution/index.ts` calls `getPerformanceLandscape()`, checks population against `MAX_POPULATION`, and asks Claude to propose mutations. The only problem is the guard on lines 96-105 that skips the entire cycle when no strategies have 30+ trades. The fix is surgical: add a `MIN_POPULATION` constant (e.g., 3) to `src/evolution/population.ts`, and when `landscape.activePaperCount < MIN_POPULATION`, bypass the 30-trade filter and instead ask Claude for *structural* mutations — new signal combinations, different entry/exit logic, fresh parameter ranges — rather than data-driven parameter tuning. This is roughly 15 lines of code in a single file, with zero new infrastructure.

The timing argument alone is decisive. The evolution cycle runs every Sunday at 18:00 UK time. If the population crashes on a Wednesday, the system waits at most 4 days and then automatically repopulates. Compare this to Approach B: the self-improvement agent also runs Sunday (19:00), but it must recognise the problem, write code, create a PR, wait for human review and merge, then wait for CI/CD to deploy, and *then* wait for the next evolution cycle to actually spawn strategies. That is a minimum two-week recovery path — one Sunday to create the PR, days to get it reviewed, and another Sunday for evolution to run with the new code. Meanwhile the system sits idle with 2 strategies accumulating almost no trade data. The VPS-side fix recovers in the same cycle it detects the problem.

The 30-trade threshold exists for good reason: you do not want to waste Sonnet tokens proposing mutations when there is no performance signal to guide them. But when population is critically low, the calculus changes entirely. Having too few strategies is a greater risk than proposing a speculative mutation. The evolution prompt (`src/evolution/prompt.ts`) already receives the full performance landscape including strategy definitions, parameters, and whatever trade data exists. Claude can reason about structural diversity — "all surviving strategies use momentum signals, try a mean-reversion variant" — even without 30 trades of statistical evidence. The prompt just needs a flag indicating this is a recovery spawn, biasing Claude toward exploration over exploitation.

Approach B (self-improvement agent) is the wrong tool for this job. The `claude.yml` workflow is designed for deep code analysis — investigating prompt quality, adjusting validator constraints, improving signal logic. It has full code access and 50 turns of Opus reasoning, costing roughly $5-10 per run. Using that heavyweight machinery to solve what amounts to "check a count and skip a guard" is like calling a structural engineer to replace a light bulb. Worse, it introduces a human-in-the-loop dependency (PR review) for what should be an autonomous recovery. The entire philosophy of this trading system is autonomous operation — the evolution cycle, the daily tournaments, the drawdown kills all happen without human intervention. Population recovery should follow the same pattern.

Approach C (both) sounds reasonable on the surface but violates a core engineering principle: a single concern should live in a single place. If both the evolution cycle and the self-improvement agent can spawn strategies in response to low population, you get overlapping responsibilities, potential race conditions (what if the self-improvement PR lands right before evolution runs?), and confusion about which system "owns" population health. The VPS evolution cycle already owns the entire strategy lifecycle — spawning, tournaments, drawdown kills, population caps. Population recovery is just the missing floor to complement the existing ceiling. Keep it in one place.

### Advocate B (GitHub Actions self-improvement)

Population collapse is a design-level failure, not a runtime hiccup. When `gap_fade_v1` got killed for 15.92% drawdown and left the system with 2 strategies that have fewer than 2 trades each, the correct response is not to loosen the 30-trade guard rail so that an under-informed Sonnet call can blindly spawn mutations from strategies with no statistical basis. The correct response is for an intelligent agent with full codebase access to diagnose *why* the population collapsed, decide what kind of strategy should replace the dead one, and commit that fix through a reviewed PR. That is exactly what the self-improvement agent already does — it runs Opus with 50 turns, SSHes into the VPS to pull the full landscape dump (which includes strategy statuses, mutation history, graduation events, and trade insights), reads the entire codebase, and creates PRs. It already has a "focus areas" section that includes investigating strategies with high drawdown and evolution proposals rejected by the validator. The only missing piece is a single paragraph in its prompt telling it to treat low population as a priority signal — a trivial change to `claude.yml`.

Approach A is dangerous precisely because it removes the guard rail that exists for a good reason. The 30-trade minimum in `runEvolutionCycle()` (line 96 of `src/evolution/index.ts`) prevents the system from making Sonnet API calls to mutate strategies that have no meaningful performance data. If you bypass this when population is low, you are asking Sonnet to propose parameter tweaks or new variants based on strategies with 0-2 trades — the resulting mutations will be random noise wearing the costume of informed evolution. Worse, these noisy children enter the population, consume paper-trading slots, and may themselves get killed by the daily tournament, creating a churn cycle where the system burns API budget spawning and killing strategies without ever accumulating the 30 trades needed for real learning. The VPS evolution cycle is fundamentally the wrong tool for this job because it operates on performance data that does not yet exist.

The self-improvement agent, by contrast, can do something the evolution cycle structurally cannot: it can *write new seed strategies*. When `dump-landscape.ts` shows a population of 2/8 with near-zero trades, the Opus agent can examine what killed the previous strategies (the graduation events and mutation history are right there in the landscape JSON), look at what market conditions the remaining strategies target, identify gaps in coverage, and commit entirely new seed strategy definitions — with fresh signal logic, parameters, and universes — as a PR. This is a qualitatively different action from parameter tweaking. It is closer to what a human portfolio manager would do: step back, assess the portfolio's structural weaknesses, and design new approaches rather than blindly mutating the survivors. The agent can also adjust thresholds (like `DRAWDOWN_KILL_PCT` or the 30-trade minimum itself) if the data suggests they are miscalibrated, but it does so through code review, not a runtime bypass.

The deployment delay concern is overstated. The self-improvement agent runs at 19:00 Sunday, one hour after evolution. If evolution skips (which it will, given the 30-trade gate), nothing is lost — the system was not going to produce useful mutations anyway. The agent creates a PR, CI runs, and the fix deploys. In practice this means new strategies are live by Monday morning before markets open. Compare this to Approach A, where a "population floor" bypass would fire at 18:00 Sunday, spawn mutations from strategies with 1-2 trades, and those mutations would then need weeks of trading to prove they are not garbage. The self-improvement path gets *better* strategies into the system faster because the Opus agent can reason about strategy design rather than blindly permuting near-empty data.

Approach C (both systems) sounds pragmatic but is actually the worst option: it adds runtime complexity to the VPS evolution code for a scenario that should be rare, creates two competing recovery mechanisms that can interfere with each other (the evolution floor spawns noisy mutations, then the self-improvement agent arrives an hour later to find slots consumed by those mutations), and doubles the maintenance surface. If population collapse is common enough to need a runtime fix, that itself is a signal the self-improvement agent should address by tuning drawdown thresholds or tournament aggressiveness — not by adding a band-aid bypass that masks the underlying problem.

### Advocate C (Both systems)

The current system has two independent failure modes, and neither Approach A nor Approach B alone addresses both. Look at what actually happened: `gap_fade_v1` was killed by the daily tournament at 15.92% drawdown, population dropped to 2/8, and both remaining strategies have fewer than 2 trades. The evolution cycle in `src/evolution/index.ts` lines 96-105 hard-skips when no paper strategies have 30+ trades — it returns immediately with a `skippedReason` string and produces zero mutations. Meanwhile, the self-improvement agent in `claude.yml` has no awareness that population health is even a concern; its prompt focuses on Sharpe ratios, drawdown, insights, and cost efficiency. Both layers are blind to the crisis in different ways. Fixing only one layer leaves the other blind.

Approach A is necessary but not sufficient. A VPS-side population floor — bypassing the 30-trade gate when `activePaperCount` falls below a minimum threshold and biasing the evolution prompt toward structural mutations — gives you immediate, autonomous self-healing. The code change is small: add a `MIN_POPULATION` constant to `population.ts`, check it before the early return, and pass a `populationCritical` flag to `buildEvolutionPrompt` so the prompt shifts toward exploration. This runs every Sunday at 18:00 without human intervention, uses existing `spawnChild`/`validateMutation` infrastructure, and can repopulate within a single cycle. Advocate A is right that this is essential. But consider what happens next: the evolution cycle spawns 2-3 structural mutations with zero trade data to guide them. These are speculative strategies born from a single Sonnet call with 4096 max tokens. If the underlying problem is systemic — the 15% drawdown kill threshold is too aggressive for current volatility, or the tournament is culling strategies before they accumulate meaningful data, or the signal expression language cannot express profitable strategies for the current regime — then the newly spawned strategies will get killed too, and the population will crash again. A-only gives you a band-aid that re-applies itself weekly but never heals the wound.

Approach B addresses root causes but fails on timing. The self-improvement agent has Opus-level reasoning, full codebase access, can SSH into the VPS to inspect logs and the database, and can make any code change — adjusting `DRAWDOWN_KILL_PCT` from 15% to 20%, adding new seed strategies with battle-tested parameters, loosening the evolution validator constraints, or even restructuring the tournament to protect young strategies from premature culling. These are changes the evolution cycle literally cannot make; it operates within the existing code and can only propose mutations within the parameter/signal/universe framework. But Advocate A correctly identifies the timing problem: the self-improvement agent creates a PR that requires merge and CI/CD deployment. Even in the best case, that is a multi-day delay. In the worst case, if no human reviews the PR before the next Sunday, you have lost an entire week. B-only means the system sits idle while waiting for a code review.

The "single concern, single place" objection Advocate A raises against Approach C mischaracterises how these two systems interact. They do not both spawn strategies in response to low population — they operate at entirely different layers. The evolution cycle is a runtime system that manages the strategy population directly: it spawns, culls, and mutates within the existing codebase. The self-improvement agent is a meta-system that modifies the codebase itself: it changes thresholds, adds seeds, improves prompts. There is no race condition because they act on different things. If the self-improvement agent adjusts `DRAWDOWN_KILL_PCT` from 15% to 18% in a PR that deploys on Tuesday, and the evolution cycle spawns two recovery strategies on Sunday, those are complementary actions — the new strategies now survive longer because the kill threshold is more forgiving. This is defence in depth, not overlapping responsibility. The evolution cycle owns runtime population management. The self-improvement agent owns code-level systemic improvements. Population health is a shared concern visible to both, just as "system reliability" is a concern shared by both application-level error handling and infrastructure-level monitoring without violating separation of concerns.

The cost objection is also overstated. Adding population awareness to the self-improvement prompt is a few lines in `claude.yml`: "If population is below 4/8, this is a critical priority. Investigate why strategies are being killed faster than created. Consider adjusting kill thresholds, adding seed strategies, or loosening evolution constraints." The self-improvement agent already SSHs into the VPS and runs `dump-landscape.ts`, which includes `activePaperCount`. It just needs to be told that a low number is alarming. This costs zero additional Opus tokens — it merely redirects attention the agent is already spending. Combined with the ~$0.05 Sonnet cost for the evolution cycle's recovery spawn, the total incremental cost is negligible. You get immediate recovery (Sunday 18:00) AND root-cause investigation (Sunday 19:00), at two different layers, for roughly the same budget as A-only.

## Round 2: Rebuttals

### Advocate A

Advocate B's "random noise" objection is the strongest argument against my position, and I want to engage with it honestly. B is correct that mutations derived from strategies with 0-2 trades lack statistical grounding — I concede that point fully. But B's conclusion does not follow. The alternative to a data-informed mutation is not "no mutation at all" — it is a *structurally-informed* mutation. The evolution prompt already receives the full strategy definitions, signal expression grammar, and parameter schemas. When I say "bypass the 30-trade gate," I do not mean "run the normal mutation logic pretending 1 trade is 30." I mean switching the prompt mode entirely: instead of asking Sonnet "which parameters should change given this performance data," you ask "given that the population has collapsed to 2 momentum-based strategies, propose a structurally different approach — mean-reversion, breakout, or a novel signal combination." This is exploration, not exploitation, and it is a perfectly valid strategy when your population is critically low. Every evolutionary algorithm has a mechanism for increasing exploration under selection pressure collapse — this is that mechanism. Will some of these exploratory spawns get killed? Certainly. But they cost ~$0.05 each in Sonnet tokens and occupy paper-trading slots that are otherwise sitting empty. The churn cycle B fears — spawn, kill, spawn, kill — is only wasteful if slots are scarce. With 2/8 occupied, we have 6 empty slots burning zero value.

B's point about the self-improvement agent writing *new seed strategies* is genuinely compelling, and I acknowledge this is something the evolution cycle cannot do in its current form. The evolution cycle operates within the existing signal/parameter/universe framework and cannot invent entirely new strategy archetypes. However, I would argue this distinction matters less than B suggests. The `spawnChild` infrastructure already supports new signal expressions, new parameter ranges, and new universe assignments — the "structural mutation" mode I am proposing is functionally equivalent to a new seed strategy, just expressed as a mutation rather than a code commit. The only strategies the self-improvement agent could write that evolution truly cannot are those requiring new code paths — a new indicator calculation, a new exit logic type, a new risk model. Those are valuable but rare, and they are not what you need in a population emergency. You need warm bodies in paper trading accumulating data, and you need them this Sunday, not next Tuesday after a PR review.

Advocate C's "band-aid" critique is partially right: if the *systemic* cause of population collapse is a miscalibrated kill threshold or an overly aggressive tournament, then repopulating without fixing that will lead to repeated crashes. I accept this. But C's own argument undermines B's position more than mine: if the root cause is systemic, then the self-improvement agent writing new seed strategies will not help either — those seeds will get killed by the same miscalibrated thresholds. The difference is that my approach gives the system *something to work with* while the systemic fix is identified. A population of 4-5 speculative strategies accumulating trade data is strictly better than a population of 2 strategies with <2 trades, because the trade data itself is what reveals whether the kill threshold is too aggressive. You cannot diagnose "strategies are being killed too fast" without strategies to observe. The VPS-side floor creates the conditions for learning; it does not pretend to be the final answer.

### Advocate B

I will concede that Advocate A and Advocate C both identify a real weakness in my Round 1 argument: the timing gap. If the self-improvement agent creates a PR on Sunday 19:00 and no human merges it until Wednesday, the system does sit idle with 2 strategies for days. That is a legitimate cost, and I understated it. However, the "two-week recovery path" framing is misleading — in practice, PRs in this project deploy via CI the moment they are merged, and a population-critical PR would be flagged as urgent. The realistic delay is hours to a day, not two weeks. More importantly, what happens during that delay? The system is not losing money — it is paper trading with 2 strategies that have fewer than 2 trades each. There is no urgency to fill slots with speculative noise. The cost of waiting a day for a well-designed strategy is far lower than the cost of immediately spawning mutations from strategies with zero statistical signal, which is what Approach A delivers.

Advocate A calls this "calling a structural engineer to replace a light bulb," but that analogy mischaracterises the problem. Replacing a light bulb is a known fix for a known failure. Population collapse after a drawdown kill is a *diagnostic* problem: why did the strategy blow up? Is the 15% kill threshold miscalibrated for current volatility? Is the tournament culling too aggressively before strategies accumulate data? Are the signal expressions in the seed pool structurally incapable of profiting in the current regime? The evolution cycle cannot answer any of these questions — it operates within fixed code and can only permute parameters. The self-improvement agent can SSH into the VPS, inspect the full landscape dump including kill history and tournament results, read the evolution validator constraints, and reason about whether the system's *configuration* is the root cause. Advocate C makes exactly this point about systemic failures, and it cuts against A, not against B. The "light bulb" is actually a wiring problem, and you do need someone who can look at the whole circuit.

On the "single concern" argument that Advocate A deploys against C but which implicitly supports B-only: I agree completely. Population health should have one owner. My position is that the self-improvement agent is the correct owner because it can address both symptoms (low population) and causes (miscalibrated thresholds, poor seed diversity, overly aggressive tournaments). The evolution cycle can only address symptoms, and it does so poorly when operating without trade data. If we accept that the self-improvement agent will eventually need population awareness anyway — and both A and C seem to agree it should — then adding a runtime bypass to the evolution cycle is not "defence in depth," it is a redundant mechanism that will spawn low-quality strategies into slots the self-improvement agent could fill with better ones an hour later.

### Advocate C

I will concede part of B's strongest point: yes, if the evolution floor spawns 3 noisy mutations at 18:00 and they consume population slots, the self-improvement agent arriving at 19:00 sees a population of 5/8 rather than 2/8, which dampens its urgency signal. That is a real coordination problem and I underplayed it in Round 1. But the fix is straightforward — tag recovery-spawned strategies with a `spawnReason: "recovery"` flag and exclude them from the population count the self-improvement agent reads, or simply have the self-improvement prompt treat any strategy with fewer than 5 trades as not meaningfully filling a slot. This is a data-labelling problem, not an architectural incompatibility. B frames it as "two competing recovery mechanisms that interfere," but they only interfere if they are unaware of each other, and making them aware costs a single field in the landscape JSON.

Advocate A's "single concern, single place" principle is sound engineering advice that I take seriously, but it is being misapplied here. The concern is not "population recovery" — it is "system health." These two mechanisms address system health at fundamentally different layers with different capabilities. The evolution cycle can spawn strategies but cannot change its own code — it cannot adjust `DRAWDOWN_KILL_PCT`, restructure tournament logic, or add new seed definitions. The self-improvement agent can change any code but cannot directly spawn strategies into the running system's database. A's principle would be violated if both systems were doing the same thing (e.g., both writing rows to the strategies table), but they are not. This is analogous to how an application has both retry logic in its HTTP client AND an alerting system that pages an engineer — both respond to failures, at different layers, and nobody calls that a separation-of-concerns violation. The evolution floor is the retry logic; the self-improvement agent is the engineer.

Where I push back hardest against both A and B is on the implicit assumption that one mechanism is *sufficient*. A's population floor will keep spawning speculative mutations every Sunday, but if the underlying problem is systemic — say the drawdown kill threshold is miscalibrated for current market volatility — those mutations will keep dying and the system enters a spawn-kill churn loop that A has no answer for. B's self-improvement agent can diagnose and fix that root cause, but it requires a human to review and merge the PR, introducing days of delay during which the system trades with a dangerously thin portfolio. Neither failure mode is hypothetical; both are visible in the current crisis. The system needs a fast responder AND a root-cause investigator. Asking which single mechanism should own this is like asking whether a hospital needs paramedics or surgeons — the answer is obviously both, operating at different timescales on the same patient.

## Round 3: Counter-rebuttals

### Advocate A

B conceded the timing gap but reframed it as "hours to a day, not two weeks" — and I want to hold that reframe to scrutiny. B's revised timeline assumes a human notices the PR promptly and treats it as urgent. But the entire point of this trading system's architecture is autonomous operation. The daily tournament kills strategies without asking permission. The drawdown monitor demotes without a PR review. The evolution cycle spawns mutations without human sign-off. Population recovery is the same class of event — an operational response to a runtime condition — and introducing a human-in-the-loop dependency for this one case is an architectural regression, not a pragmatic compromise. B says "the system is not losing money" during the delay because it is paper trading, but that misses the real cost: it is losing *learning time*. Every day with 2 idle strategies is a day where the system accumulates no trade data, which means the next evolution cycle also skips, which means the self-improvement agent's next run also sees a barren landscape. The delay compounds.

C's proposal to tag recovery strategies with `spawnReason: "recovery"` and exclude them from the self-improvement agent's population count is technically clean, but I want to point out what it reveals: C is building coordination machinery between two systems to prevent them from interfering with each other. That is complexity born from having two systems doing overlapping work. If the evolution floor is the sole recovery mechanism, no tagging is needed, no "exclude from count" logic, no prompt engineering to tell the self-improvement agent to ignore strategies with fewer than 5 trades. C is inadvertently proving my point about single-concern ownership — the coordination overhead is a code smell indicating that two systems are stepping on each other's territory. That said, I want to be honest: C's underlying argument about different layers is not wrong. The self-improvement agent genuinely can do things the evolution cycle cannot — adjusting kill thresholds, restructuring tournament logic, adding new indicator calculations. I am starting to see that population *awareness* in the self-improvement agent is valuable, not as a recovery mechanism, but as a diagnostic signal. If the agent sees population repeatedly crashing and recovering via the evolution floor, that pattern should trigger it to investigate root causes. The distinction I am drawing is between the self-improvement agent as a *responder* to low population (B's position, which I reject) and as an *observer* of population health trends (which I think has merit). Let the evolution floor handle the immediate crisis autonomously. Let the self-improvement agent notice if the crisis keeps recurring and fix the systemic cause. That is genuine defence in depth without overlapping responsibility.

### Advocate B

I will be honest: Advocate A's Round 2 reframing of recovery spawns as "structural exploration rather than data-driven exploitation" is more compelling than I initially gave credit for. The distinction between asking Sonnet to tweak parameters on 1-trade strategies versus asking it to propose a structurally different archetype (mean-reversion when the population is all momentum) is real, and I was wrong to dismiss it as "random noise in a costume." Exploration under population pressure is a legitimate evolutionary strategy. However, this concession actually strengthens my core objection rather than weakening it. A's best-case scenario is that the evolution cycle spawns a structurally novel strategy that then needs weeks of paper trading to prove itself. During those weeks, the system still cannot run meaningful evolution (the new strategy has <30 trades), so you have recovered *population count* without recovering *evolutionary capacity*. The system looks healthier on the dashboard but remains functionally stuck. The self-improvement agent, by contrast, can address the actual bottleneck: it can lower the 30-trade threshold for young strategies, add tournament protection for strategies under 10 trades, or adjust the drawdown kill percentage — fixes that restore the system's ability to learn, not just its headcount.

Advocate C's tagging proposal (marking recovery spawns with `spawnReason: "recovery"` so the self-improvement agent ignores them in its population count) is clever but reveals the deeper problem with Approach C. Every coordination mechanism between the two systems is another piece of bespoke logic that must be maintained, tested, and reasoned about. First you need the tag. Then the self-improvement prompt needs to understand the tag. Then you need to decide: does the evolution cycle count tagged strategies toward its own population floor on subsequent Sundays? What if a recovery-spawned strategy accumulates 30+ trades and becomes legitimate — does it lose the tag? What if the self-improvement agent deploys a PR that adds new seeds while tagged recovery strategies are still active — do you now exceed MAX_POPULATION? Each question has an answer, but the accumulation of answers is exactly the kind of coordination complexity that Advocate A rightly warns against in distributed systems. A single owner — the self-improvement agent — eliminates this entire class of problem.

A makes the interesting argument that recovery spawns "generate diagnostic trade data even if they die," creating the conditions for the system to learn whether kill thresholds are miscalibrated. I concede this has some merit — you do need strategies in the system to observe failure patterns. But the self-improvement agent does not need *new* strategies to perform this diagnosis. The system already has the full kill history of `gap_fade_v1` (15.92% drawdown, the tournament that killed it, the market conditions at the time) and the near-zero trade counts of the two survivors. That data is already in the landscape dump. The Opus agent can reason about whether 15% is too aggressive a kill threshold by examining the dead strategy's trade history and comparing it to ATR-scaled expected drawdowns — it does not need to sacrifice additional strategies to learn this. The "generate data by spawning cannon fodder" approach is empiricism where analysis would suffice, and it costs weeks of elapsed time that the analytical approach does not.

### Advocate C

After two rounds, the debate is converging toward Approach C more than either A or B may realise. Advocate A now explicitly concedes that speculative spawns are "not the final answer" and that systemic root causes (miscalibrated kill thresholds, overly aggressive tournaments) require fixes the evolution cycle cannot make. That is an endorsement of the self-improvement layer. More tellingly, A's Round 3 closing paragraph arrives at a position that *is* Approach C: "Let the evolution floor handle the immediate crisis autonomously. Let the self-improvement agent notice if the crisis keeps recurring and fix the systemic cause." A calls this "genuine defence in depth without overlapping responsibility" — which is exactly the framing I used in Round 1. The only remaining disagreement is whether the self-improvement agent should actively spawn seed strategies (B's position) or merely observe and adjust thresholds (A's concession). Meanwhile, B conceded the timing gap is real and costly, acknowledged that trade data generation during the delay matters, and implicitly accepted that the evolution floor provides value as a data-generating mechanism even if its spawns are statistically uninformed. Both advocates have moved toward C; they are debating the *scope* of C, not whether C is correct.

The minimal version of C that satisfies both A and B is narrower than my Round 1 proposal but preserves the essential two-layer structure. First, the evolution floor is conservative: spawn at most 2 recovery strategies per cycle (not filling all 6 empty slots), tag them with `spawnReason: "population_recovery"`, and bias the prompt toward structural diversity rather than parameter mutation. This directly addresses B's "noisy churn" concern — 2 strategies cannot consume the population, leaving 4 slots for whatever the self-improvement agent designs. A's objection about coordination overhead ("tagging is complexity born from overlapping work") is fair but overstated — a single enum field in the landscape JSON is not architectural complexity; it is metadata. Second, the self-improvement prompt change is one paragraph in `claude.yml` telling Opus to treat `activePaperCount < 4` as a critical signal and to investigate kill/tournament history before proposing threshold adjustments or new seeds. A is right that the agent's primary role here should be diagnostic — observing whether recovery spawns keep dying and fixing the systemic cause — but B is also right that Opus can write qualitatively better seed strategies than Sonnet can improvise under a 4096-token limit. Both roles coexist without conflict: Opus diagnoses, and if it determines new seeds are the answer, it writes them; if it determines thresholds are the answer, it adjusts those instead. Third, the self-improvement agent should be empowered to tune or disable the population floor itself. If it observes repeated recovery-spawn-then-kill cycles, it can raise `MIN_POPULATION` thresholds, adjust the spawn cap, or even remove the floor entirely if the root cause is better addressed structurally. This makes the floor self-correcting rather than permanent.

A's Round 3 concern about "coordination machinery" proving that two systems are stepping on each other is the last substantive objection, and I think A actually answered it themselves: the evolution floor owns runtime spawning, the self-improvement agent owns code-level systemic fixes, and population health is a shared *signal* visible to both — not a shared *action*. The tagging mechanism is not two systems coordinating to avoid collision; it is one system (evolution) leaving a breadcrumb that the other system (self-improvement) can read for diagnostic context. This is no different from the evolution cycle reading `graduationEvents` that the graduation gate produces, or the self-improvement agent reading `mutationHistory` that evolution produces. The system already has cross-layer observability; adding one more field does not change the architecture. What we are converging on is not a compromise but a recognition that population health, like system health generally, is a concern that manifests at multiple layers and requires responses at each layer appropriate to that layer's capabilities. The evolution floor responds fast with what it has. The self-improvement agent responds thoughtfully with what it can change. Neither substitutes for the other.

## Round 4: Synthesis

### Advocate A

The debate has converged more than I expected. All three advocates now agree on the core points: (1) the VPS-side evolution floor is necessary for immediate, autonomous recovery -- no one is still arguing the system should sit idle with 2/8 strategies waiting for a PR review; (2) the self-improvement agent should have population awareness as a diagnostic signal; (3) the 30-trade gate should be bypassed under population pressure, but the resulting spawns should be structurally exploratory rather than data-driven parameter mutations. The remaining contention is scope: should the self-improvement agent actively write seed strategies and adjust thresholds in response to low population (B and C's position), or should it limit itself to observing whether recovery spawns succeed and only intervening when it detects a recurring pattern of spawn-kill churn (my revised position from Round 3)? B also maintains that the self-improvement agent should be the sole owner, but this position has weakened -- B conceded the timing gap is real and acknowledged that recovery spawns generate valuable diagnostic data, which implicitly accepts the evolution floor's utility.

The coordination overhead question -- tagging, slot counting, prompt awareness -- is the last genuinely unresolved design issue.
